1. 架构概述

AgentFlow 是一个以多轮对话为核心的智能体工作流编排引擎。它将对话过程建模为有向图（DAG），每一轮用户输入都会触发图的执行引擎，通过意图识别 → 路由决策 → 节点执行 → 上下文更新的闭环完成业务流程。

核心设计原则

  • 对话即流程：每一轮对话都是一次图的遍历，用户的意图决定路径选择
  • 精准路由：基于 LLM 的意图分类器 + 规则引擎双保险，确保路由准确率
  • 上下文感知：全局 Memory + 局部 Slot 双层上下文，确保多轮衔接自然
  • 人机协同：关键节点支持人工确认，流程可随时暂停和恢复
  • 可观测性：每个节点的输入/输出/耗时/状态全链路记录

2. 节点定义

AgentFlow 将业务流程建模为有向图，每个节点代表一个处理单元。节点之间通过边（Edge）连接，边上可附加条件表达式。

▶

开始节点 StartNode

工作流入口。负责初始化会话上下文、加载用户信息、设置初始 Slot。每个工作流有且仅有一个开始节点。

■

结束节点 EndNode

工作流出口。执行收尾动作（汇总结果、发送通知、清理资源）。支持条件结束，可输出结构化结果。

⬡

智能体节点 AgentNode

核心执行单元。绑定一个具体的 Agent（含 System Prompt、模型配置、工具集）。接收上下文输入，生成 LLM 响应。

◇

路由节点 RouterNode

条件分支决策。根据意图识别结果或表达式，选择下一条执行路径。支持多分支、默认分支和动态路由。

☺

人工确认节点 HumanNode

流程暂停等待人工审批。支持超时策略、自动提醒、审批/拒绝分支。流程状态持久化到检查点。

◈

评估节点 EvalNode

对智能体输出进行质量评估。支持 LLM-as-Judge、规则校验、指标评分。评估结果可触发重试或路由。

⊞

工作流节点 WorkflowNode

嵌套子流程。引用另一个 AgentFlow 工作流，实现流程复用和组合编排。支持参数传递和结果映射。

⧉

插件节点 PluginNode

外部能力扩展。调用 HTTP API、数据库、消息队列等外部服务。支持输入映射、输出转换和错误重试。

3. 多轮对话引擎

AgentFlow 的核心是一个对话驱动的执行引擎。每一轮用户输入都会触发一次完整的"感知 → 思考 → 行动"循环。

3.1 单轮对话处理流程

用户消息 → 消息预处理 → 上下文组装 → 意图识别 → 路由决策 → 节点执行 → 结果输出 → 上下文更新

3.2 多轮状态机

每轮对话结束后，引擎会更新会话状态并等待下一轮输入。会话状态机如下：

enum SessionState {
  ACTIVE           // 正常对话中，等待用户输入
  WAITING_HUMAN    // 等待人工确认（暂停）
  WAITING_INPUT    // 等待用户补充信息（追问）
  COMPLETED        // 流程已完成
  FAILED           // 流程异常终止
  SUSPENDED        // 手动挂起
}

interface Session {
  id: string
  workflowId: string
  state: SessionState
  currentNodeId: string           // 当前停留的节点
  turnCount: number                // 已对话轮次
  slots: Record<string, any>      // 当前会话的槽位数据
  history: Turn[]                   // 对话历史
  intentHistory: IntentResult[]   // 意图识别历史
  checkpoints: Checkpoint[]        // 检查点（用于恢复）
  createdAt: Date
  updatedAt: Date
}

3.3 对话轮次数据结构

interface Turn {
  turnIndex: number
  timestamp: Date
  user: {
    message: string                // 用户原始消息
    attachments?: Attachment[]     // 附件（图片、文件等）

}

4. 意图识别与路由

这是 AgentFlow 最核心的模块。精准的意图识别是正确路由的前提，而正确的路由决定了对话流程的走向。

4.1 三层意图识别架构

设计思路：采用"规则层 → LLM 层 → 融合层"三级架构，规则层处理确定性场景（快且便宜），LLM 层处理模糊场景（慢但准），融合层做最终决策。

第一层：规则引擎（确定性匹配）

interface RuleEngine {
  rules: IntentRule[]
}

interface IntentRule {
  intent: string
  patterns: string[]          // 正则表达式
  keywords: string[][]        // 关键词组合（AND 关系）
  slotExtractors?: SlotExtractor[]  // 槽位提取器
  confidence: number          // 匹配时的置信度（通常 0.95+）
  priority: number            // 规则优先级
}

// 示例规则
{
  "intent": "query_order",
  "patterns": [
    "查.*订单",
    "订单.*?(\\d+)",
    "我的.*快递"
  ],
  "keywords": [
    ["订单"],
    ["查", "快递"],
    ["物流", "进度"]
  ],
  "slotExtractors": [{
    "slot": "orderId",
    "pattern": "(?:订单号?|单号)[：:]?\\s*(\\w+)"

  "priority": 1
}

第三层：融合决策

function fuseIntentResults(
  ruleResult: IntentResult | null,
  llmResult: IntentResult,
  context: ConversationContext
): FinalIntent {

  // 1. 规则命中高置信度 → 直接采用
  if (ruleResult && ruleResult.confidence >= 0.9) {
    return { ...ruleResult, source: "rule" }
  }

  // 2. 上下文修正：如果 LLM 结果与上一轮意图矛盾，降权
  let adjustedConfidence = llmResult.confidence
  if (context.lastIntent && isContradictory(llmResult, context.lastIntent)) {
    adjustedConfidence *= 0.7
  }

  // 3. 上下文增强：如果 LLM 结果是上一轮的延续，加权
  if (context.lastIntent && isContinuation(llmResult, context.lastIntent)) {
    adjustedConfidence = Math.min(adjustedConfidence + 0.15, 1.0)
  }

  // 4. 置信度达标 → 采用 LLM 结果
  if (adjustedConfidence >= 0.7) {
    return { ...llmResult, confidence: adjustedConfidence, source: "llm" }
  }

  // 5. 低置信度 → 触发澄清追问
  return {
    intent: "unclear",
    confidence: adjustedConfidence,
    candidates: [ruleResult, llmResult].filter(Boolean),
    clarification: generateClarificationQuestion([ruleResult, llmResult])
  }
}

4.2 路由决策引擎

路由决策基于意图识别结果，决定下一步执行哪个节点。

class RouterEngine {
  decide(intent: FinalIntent, session: Session, graph: Graph): RouteDecision {

    // Case 1: 意图明确 → 匹配路由表
    if (intent.intent !== "unclear") {
      const route = this.matchRoute(intent, graph.currentNode.edges)
      if (route) return {
        target: route.target,
        reason: `意图 ${intent.intent} 匹配路由规则`,
        confidence: intent.confidence
      }
    }

    // Case 2: 意图不明确 → 检查是否需要追问
    if (intent.clarification) {
      return {
        target: "__clarification__",  // 内置追问节点
        message: intent.clarification,
        reason: "意图不明确，需要追问用户",
        confidence: intent.confidence
      }
    }

    // Case 3: 话题切换检测
    if (this.isTopicSwitch(intent, session)) {
      return {
        target: this.findEntryPoint(intent, graph),
        reason: "检测到话题切换，跳转到新流程入口",
        topicSwitch: true
      }
    }

    // Case 4: 兜底路由
    return {
      target: graph.currentNode.defaultEdge?.target ?? "__fallback__",
      reason: "使用默认路由",
      confidence: 0
    }
  }

  // 话题切换检测
  isTopicSwitch(intent: FinalIntent, session: Session): boolean {
    const recentIntents = session.intentHistory.slice(-3)
    const sameTopic = recentIntents.every(i => i.topicGroup === intent.topicGroup)
    return !sameTopic && intent.confidence > 0.8
  }
}
关键设计：当检测到用户话题切换时，引擎会自动保存当前流程的检查点（Checkpoint），然后跳转到新话题的流程入口。当新话题处理完毕后，可以选择性地恢复之前的流程。

5. 上下文管理

精准的意图识别和自然的多轮衔接，都依赖于完善的上下文管理体系。AgentFlow 采用三层上下文架构。

5.1 三层上下文

层级作用域生命周期存储内容
全局 Memory 用户级别 永久 用户画像、偏好、长期记忆
会话 Context 会话级别 会话期间 对话历史、当前状态、检查点
节点 Slot 节点级别 节点执行期间 节点输入参数、中间结果、临时变量

5.2 上下文组装策略

每轮对话执行前，引擎会将三层上下文组装成统一的 Context Object，供节点使用。

interface AssembledContext {
  // 全局记忆
  memory: {
    userProfile: UserProfile
    preferences: Record<string, any>
    longTermMemory: MemoryEntry[]    // 经 LLM 摘要的长期记忆
  }

  // 会话上下文
  session: {
    id: string
    state: SessionState
    currentNode: string
    turnCount: number
    slots: Record<string, any>       // 当前会话所有槽位
    history: Turn[]                    // 完整对话历史
    recentHistory: Turn[]              // 最近N轮（用于LLM）
    summarizedHistory: string          // 更早历史的摘要
    intentHistory: IntentResult[]     // 意图识别轨迹
  }

  // 当前输入
  currentTurn: {
    userMessage: string
    timestamp: Date
    metadata: Record<string, any>     // 渠道、设备等元信息
  }
}

5.3 上下文更新机制

class ContextUpdater {
  // 每轮对话结束后调用
  async update(session: Session, turn: Turn) {
    // 1. 更新槽位（从节点输出中提取）
    this.updateSlots(session, turn)

    // 2. 追加对话历史
    session.history.push(turn)

    // 3. 追加意图历史
    session.intentHistory.push(turn.system.intent)

    // 4. 历史压缩（超过阈值时摘要化）
    if (session.history.length > 20) {
      session.summarizedHistory = await this.summarizeHistory(
        session.history.slice(0, -10)
      )
      session.history = session.history.slice(-10)
    }

    // 5. 更新全局记忆（重要信息持久化）
    if (turn.system.slotsUpdated["userPreference"]) {
      await this.updateMemory(session.userId, turn)
    }

    // 6. 创建检查点
    session.checkpoints.push({
      turnIndex: turn.turnIndex,
      snapshot: deepClone(session),
      timestamp: new Date()
    })
  }
}

6. 执行引擎

6.1 核心执行循环

class AgentFlowEngine {

  async handleMessage(sessionId: string, userMessage: string): Promise<EngineResult> {

    // ① 加载会话
    const session = await this.sessionStore.load(sessionId)

    // ② 组装上下文
    const context = await this.contextAssembler.assemble(session, userMessage)

    // ③ 意图识别
    const intent = await this.intentClassifier.classify(context)

    // ④ 路由决策
    const route = this.router.decide(intent, session, this.graph)

    // ⑤ 执行节点链
    const executions: NodeExecution[] = []
    let currentNode = this.graph.getNode(route.target)

    while (currentNode && currentNode.type !== "end") {
      const execution = await this.executeNode(currentNode, context, session)
      executions.push(execution)

      // 节点执行后可能产生新的路由（链式执行）
      if (execution.nextNodeId) {
        currentNode = this.graph.getNode(execution.nextNodeId)
      } else {
        break
      }

      // 人工确认节点 → 暂停，等待回调
      if (currentNode?.type === "human") {
        session.state = "WAITING_HUMAN"
        session.currentNodeId = currentNode.id
        break
      }
    }

    // ⑥ 构建响应
    const response = this.buildResponse(executions, route)

    // ⑦ 更新上下文
    const turn = this.buildTurn(session, userMessage, intent, route, executions, response)
    await this.contextUpdater.update(session, turn)

    // ⑧ 持久化
    await this.sessionStore.save(session)

    return { response, session }
  }

  // 执行单个节点
  async executeNode(node: Node, ctx: AssembledContext, session: Session): Promise<NodeExecution> {
    const startTime = Date.now()

    try {
      const handler = this.nodeHandlers[node.type]
      const input = this.resolveInputMapping(node, ctx)

      const output = await Promise.race([
        handler.execute(input, node.config, ctx),
        timeout(node.timeout ?? 30000)
      ])

      return {
        nodeId: node.id,
        status: "success",
        input, output,
        duration: Date.now() - startTime,
        nextNodeId: this.resolveNextNode(node, output)
      }
    } catch (error) {
      return {
        nodeId: node.id,
        status: "error",
        error: error.message,
        duration: Date.now() - startTime,
        nextNodeId: node.onError
      }
    }
  }
}

6.2 节点执行链

一轮对话中可能连续执行多个节点。典型的执行链如下：

示例：用户说"帮我查一下订单 12345 的物流"

RouterNode (识别意图: query_order) → PluginNode (调用订单API) → AgentNode (组织回复语言) → EvalNode (检查回复质量) → 输出给用户

6.3 检查点与恢复

interface Checkpoint {
  turnIndex: number
  timestamp: Date
  snapshot: {
    currentNodeId: string
    slots: Record<string, any>
    state: SessionState
  }
}

// 从检查点恢复（用于话题切换后回到之前的流程）
async restoreFromCheckpoint(session: Session, turnIndex: number) {
  const cp = session.checkpoints.find(c => c.turnIndex === turnIndex)
  if (cp) {
    session.currentNodeId = cp.snapshot.currentNodeId
    Object.assign(session.slots, cp.snapshot.slots)
    session.state = cp.snapshot.state
  }
}

  // 边定义（节点之间的连接）
  edges: Edge[]

  // 全局配置
  config: {
    maxTurns: number               // 最大对话轮次
    sessionTimeout: number          // 会话超时（ms）
    defaultModel: string            // 默认使用的模型
    intentClassifierModel: string   // 意图分类器模型
    contextWindowSize: number      // 上下文窗口大小
  }
}

interface Edge {
  id: string
  source: string                   // 源节点 ID
  target: string                   // 目标节点 ID
  condition?: string               // 条件表达式（可选）
  label?: string                   // 边的标签
  priority?: number                // 优先级（多条件匹配时）
}

9.2 多轮对话示例

// 轮次 1: 用户问订单
User: "我的订单 12345 到哪了？"
→ 规则引擎命中: pattern="订单.*(\d+)" → intent=query_order, confidence=0.95
→ 路由: order-query-agent
→ PluginNode: GET /api/orders/12345 → { status: "配送中", eta: "明天" }
→ AgentNode: "您的订单 12345 正在配送中，预计明天送达。需要查看物流详情吗？"
→ Slots: { orderId: "12345", orderStatus: "配送中" }

// 轮次 2: 上下文延续（省略了orderId）
User: "我想退款"
→ LLM分类器: intent=refund, confidence=0.88
→ 上下文增强: 检测到 slots.orderId 已存在 → confidence=0.95
→ 路由: refund-workflow (携带 orderId=12345)
→ WorkflowNode: 启动退款流程
→ HumanNode: "退款金额 ¥299，请确认" → WAITING_HUMAN

// 轮次 3: 人工确认回调
Manager: approve
→ HumanNode → process-refund
→ PluginNode: POST /api/refunds → { refundId: "R789", status: "处理中" }
→ AgentNode: "退款已提交，退款单号 R789，预计 3 个工作日到账。还有其他问题吗？"

// 轮次 4: 话题切换
User: "对了，你们最近有什么优惠活动？"
→ LLM分类器: intent=promotion_query, confidence=0.82
→ 话题切换检测: 与之前 refund 话题不同 → 保存检查点，跳转新流程
→ 路由: general-agent
→ AgentNode: "目前有满200减30的活动..."
→ Slots: { topic: "promotion" }

// 轮次 5: 回到之前话题（可选）
User: "之前那个退款到账了吗？"
→ LLM分类器: intent=refund_status_query, confidence=0.85
→ 上下文关联: 检测到历史 slots.refundId=R789
→ 路由: order-query-agent (携带 refundId)
→ AgentNode: "退款 R789 已于今天到账，请查收。"

9.3 关键设计要点总结

精准路由的核心策略：
  • 规则优先：确定性场景用正则/关键词直接命中，避免 LLM 调用开销
  • 上下文修正：利用历史意图和已填充的 Slot 修正当前意图的置信度
  • 低置信度追问：当置信度 < 0.7 时，不强行路由，而是追问用户澄清
  • 话题切换感知：检测用户是否切换了话题，自动保存/恢复检查点
  • Slot 继承：新话题路由时，自动携带相关的 Slot（如 orderId），减少用户重复输入
